# ws npm package CVE in wasm toolchain yarn.lock (GHSA-58qx-3vcg-4xpx)

**Issue:** [krill-oss#131](https://github.com/Sautner-Studio-LLC/krill-oss/issues/131)
**Root cause category:** Dependency hygiene — CVE in npm toolchain lockfile not auto-updated by Gradle
**Module:** `krill-sdk`

## What happened

The Kotlin/Wasm compiler generates a `yarn.lock` at
`krill-sdk/kotlin-js-store/wasm/yarn.lock` that pins the npm packages used by
the Webpack/Karma wasm JS build toolchain. The `ws` package was pinned to
8.18.3, which falls in the vulnerable range (≥ 8.0.0, < 8.20.1) for
GHSA-58qx-3vcg-4xpx (Denial of Service via malformed HTTP upgrade headers).

`ws` is a dev-only toolchain dependency — it is **not** shipped in the
published `krill-sdk` artifact and carries no runtime risk for consumers.
However, Dependabot and the Nightly Dependency Scan still flag it because
the lock file lives in the repo.

## Fix

- `krill-sdk/kotlin-js-store/wasm/yarn.lock`: updated `ws` entry from 8.18.3
  to 8.20.1, including the resolved URL and sha512 integrity hash sourced
  directly from the yarnpkg registry.
- `krill-sdk/build.gradle.kts`: bumped patch from 0.0.46 to 0.0.47 (required
  on every `krill-sdk/**` change per project rules).

## Prevention

- **Regenerate the wasm yarn.lock periodically or after Gradle wrapper
  updates.** The lock file is auto-generated by the Kotlin/Wasm Gradle plugin;
  re-running `./gradlew wasmJsTest` (or any wasm task) will refresh it when
  the plugin's bundled npm dep ranges are updated in a newer Kotlin release.
- **Classify npm lockfiles in `kotlin-js-store/` as toolchain-only.**
  Vulnerabilities in these packages carry no production risk but should still
  be cleared promptly to keep Dependabot signal clean for genuinely
  runtime-affecting alerts.
- **Verify integrity hashes before committing manual lockfile edits.** The
  sha512 for the patched tarball was confirmed by downloading from both
  `registry.yarnpkg.com` and `registry.npmjs.org` and checking they match.
