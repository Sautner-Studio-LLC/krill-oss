## Routing an upstream root cause (dev → dev)

When investigation bottoms out in the **other** repo:

1. **Don't fix locally.** Leave the originating issue open; only mitigate downstream if QA explicitly accepts it (with a tracker describing the workaround).
2. **File upstream:** `gh issue create --repo <other-repo> --assignee krill-blue-bot ...` with labels passed explicitly. **No `qa` / `qa-agent` umbrella** — that's QA's signal; `--assignee` is what gets the upstream dev. Reuse the originating QA report's body shape.
3. **Cross-link both ways:** comment on origin *"Routed upstream as `<repo>#<m>`. Leaving open until that lands."*; on upstream body add `Observed downstream:` pointing back.
4. **`dev-krill` only:** also open a tracker here via `dev-cross-repo.yml`.
5. **Wait** — only Ghost closes QA issues after re-verify.

## Read the issue

```bash
gh issue view <n> --repo <my-repo> --comments   # body + comments
```

If you can't reproduce after a real attempt, comment asking Ghost (or Ben) for the missing detail and stop. Do not fix what you cannot reproduce.

## Acknowledge and report progress

Issue thread is the only place Ben (and Ghost) can see what you're doing — treat it as a status log.

**On pick-up:** apply `status:in-progress` (create the label once with `gh label create status:in-progress --color BFD4F2` if missing — it auto-clears when the PR's `Closes #<n>` lands). Comment a one-line plan: *"Picking this up — \<approach\>"* (or *"investigating reproducer"* if no approach yet).

**During work — comment at inflection points:** introducing commit (SHA + title), choice between two viable approaches, routing root cause upstream, unexpected blocker that delays/rescopes, PR opened (link it). Don't narrate every command; comment on decisions and pivots only.

## Per-issue work loop

1. **Reproduce locally first.** If you can't, ask in-thread and stop. (`dev-krill-agents` exception: most issues are doc/workflow gaps — verify by running `scripts/build-claude-md.sh <role> -` and reading as a fresh agent would.)
2. **Locate the regression.** `git log -S '<token>'` + `git blame`. Capture the introducing SHA + title for the PR body.
3. **Branch.** `fix/<issue#>-<slug>` from `main` (slug = 3–5 hyphens). `feat/<slug>` only on explicit Ben ask; `chore/<slug>` for repo plumbing or `krill-agents` doc/workflow.
4. **Test-first.** Failing test → make it pass. If genuinely untestable (UI on real device, hardware), say so in the PR — no token assertions. Doc-only: a markdown sanity test guarding against the regression. (`krill-agents` has no test framework.)
5. **Test location:** see `build-krill.md` / `build-krill-oss.md` per module.
6. **Lessons entry** — `docs/lessons/YYYY-MM-DD-<slug>.md` on `krill` and `krill-oss` (sections: *What happened*, *Fix*, *Prevention*; schema in `docs/lessons/README.md`). **CI hard-fails the PR if missing.** Not required on `krill-agents`.
7. **Open the PR** via `.github/PULL_REQUEST_TEMPLATE.md` with `Closes #<n>`. Decide on `needs-qa-verify` (see below). **Never add `needs-qa-verify` on `krill-agents`** — no listener.
8. **Hand off (krill / krill-oss).** With `needs-qa-verify`: comment `@qa-agent please verify` + reproduction script (see `cross-repo-protocol.md`). Without: `@` Ben directly with a one-line reason ("doc-only — no runtime change"). On `krill-agents`: `@` Ben.
9. **Stop.** Wait for Ghost **PASS** (if labeled) + Ben's approval; else just Ben's approval.

### When to apply `needs-qa-verify` (krill / krill-oss only)

`needs-qa-verify` triggers `Verify Agent Ghost.yml` — builds 4 debs, installs on ghost runner, full smoke+regression sweep. Real cost; don't pay for PRs that can't move runtime.

**Apply when the PR could change runtime behavior or what ships in the deb:**
- Production source: `server/`, `shared/`, `composeApp/`, `androidApp/`, `iosApp/`, `ksp/` (krill); `krill-sdk/`, `krill-mcp/server/`, `krill-pi4j/` (krill-oss).
- Build/Gradle/CI/packaging (`build.gradle.kts`, `libs.versions.toml`, debian packaging, GH Actions YAML that builds/installs).
- KSP processor changes (regenerate shipped code).
- MCP tool-shape or skill-instruction changes on `krill-oss` — agents consume at runtime.
- Anything you're unsure about. When in doubt, label it.

**Skip when no runtime risk:** pure docs (`*.md`, `docs/**`, `docs/lessons/` alone), comment-only edits, `openspec/`, repo plumbing the build doesn't consume (`.gitignore`, `CODEOWNERS`, `.github/ISSUE_TEMPLATE/`).

Mixed PR (code fix + lessons entry): the code drives the call — apply the label. State your call in the PR's Test plan section so Ben can sanity-check it.

## Writing tests that don't flake

CI runs on hardware that doubles as a real Krill server — production paths (`/etc/krill/...`, `~/.krill/install_id`, `/srv/krill/...`, `/var/lib/krill/...`) exist with real PIN files and install ids, alongside background traffic. Tests that assume otherwise hard-fail.

If you can't satisfy the rules below, the unit-under-test is wrong-shaped — parameterise the seam, or skip the test and say why in the PR.

### Bans (hard fails)

- **No `delay()` / `Thread.sleep` for time passing.** Use `runTest { … }` + `advanceTimeBy(...)` / `advanceUntilIdle()` from `kotlinx-coroutines-test` (alias `kotlin-coroutines-test` in `libs.versions.toml`; add `testImplementation(libs.kotlin.coroutines.test)` if not yet on the module classpath).
- **No wall-clock-relative concurrency assertions** ("fires exactly once after 250 ms"). If the invariant is a mutex/serialisation, assert it without time (e.g. `Semaphore.tryAcquire()` probe) or feed ordered events to a virtual scheduler.
- **No reading/writing real production paths** (`/etc/krill/credentials/pin_derived_key`, `/srv/krill/data/krill_v3.db`, `~/.krill/install_id`, `/var/lib/krill/trusted`). Inject the path (or its resolved value) as a parameter; pass a fixture string in the test.
- **No network calls, even localhost** (no Ktor `testApplication { … }` binding a real port). Use `MockEngine` against the route handler factory, or call the helper the route delegates to.
- **No deps on `installId()`, `hostName`, `SystemInfo.isServer()`, or other platform `expect-val`.** Refactor the unit to take the value as a parameter; resolve once at the boundary. `peerProjection(nodes, selfId)` (from `krill#190`) is the canonical shape.
- **No `@BeforeEach { File(...).parentFile?.mkdirs() }`** — that's a missing seam.

### Positive patterns

- **Parameterise the platform seam:** helper takes `selfId: String` / `now: () -> Long` / `clock: Clock`; caller resolves once.
- **Stub via `open` + subclass** (e.g. `PinProvider` overridden in test).
- **Virtual time:** `runTest { advanceTimeBy(150.milliseconds); coroutine.cancelAndJoin() }`.
- **`MockK` for collaborator boundaries, not the unit under test.**
- **Prove with a hostile env:** `HOME=/nonexistent ./gradlew :module:test` should still pass.

Tests that genuinely need a real environment (integration, Pi hardware) don't belong in default `:test`; tag and exclude them, or skip writing and flag the gap.

## Conventions

- **Branch:** `fix/<issue#>-<slug>` (QA bugs), `chore/<slug>` (repo plumbing / `krill-agents` docs), `feat/<slug>` (explicit Ben ask only).
- **Commit subject:** `<type>(<scope>): <imperative> (#<issue>)`, type ∈ `fix|docs|chore|feat|refactor|test`. Body: *why* in 2–3 sentences. Keep the `Co-Authored-By:` trailer.
- **PR title** mirrors commit subject; body follows `.github/PULL_REQUEST_TEMPLATE.md`; always include `Closes #<n>`.
- **Re-tagging:** if QA mislabeled, comment first, then re-tag. Don't drop labels silently.

## Hard rules — don't violate

- **Never** edit outside your owned repo. Cross-repo work goes through GitHub issues.
- **Never** work an issue not assigned to `krill-blue-bot`.
- **Never** start fix work silently — first action is the ack comment + `status:in-progress`. Surface decisions at inflection points; Ben can't read your shell.
- **Never** merge your own PR. Never close a QA issue without Ghost **PASS**. Never skip the `docs/lessons/` entry.

- **Never** add `needs-qa-verify` on `krill-agents` (no listener) or reflexively on no-risk `krill` / `krill-oss` PRs (see apply-vs-skip rule above).

CI (`.github/workflows/release-sdk.yml`) auto-dispatches the upstream Maven publish in `Sautner-Studio-LLC/krill` on push to `main` — no manual republish. **Skipping the bump silently leaves the fix out of Maven Central** (build still passes). Dispatch needs `KRILL_PUBLISH_TOKEN` (PAT scoped to `Sautner-Studio-LLC/krill` with `Actions: read+write`, `Contents: read`).

**Consumer pin:** `krill-mcp/gradle/libs.versions.toml` `krill-sdk = "x.y.z"` — only bump in a follow-up PR **after** the new artifact is live, or krill-mcp build breaks.

## `krill-mcp/` — JDK 21, JVM-only

```bash
cd krill-mcp
./gradlew :krill-mcp-service:shadowJar                  # → krill-mcp-service/build/libs/krill-mcp-all.jar
./gradlew :krill-mcp-service:build                      # depends on shadowJar + tests
./gradlew :krill-mcp-service:test --tests "FQCN.method"
java -jar krill-mcp-service/build/libs/krill-mcp-all.jar  # run; needs /etc/krill-mcp/config + credentials
```

`./gradlew :krill-mcp-service:run` is **not wired** — use the fat JAR.

Daemon expects `/etc/krill-mcp/config.json` (`KrillMcpConfig`: `listenPort`, `pinDerivedKeyPath`, `seeds[]`) + `/etc/krill-mcp/credentials/pin_derived_key` (64-hex HMAC of cluster PIN). Without the key, server boots but `/mcp` returns 401; `/healthz` stays unauthenticated.

### Architecture

```
Claude ──Bearer──▶ KtorApp /mcp ──▶ McpServer ──▶ Tool.execute() ──▶ KrillRegistry.resolve() ──▶ KrillClient ──HTTPS──▶ Krill server
```

Modules under `krill-mcp-service/src/main/kotlin/krill/zone/mcp/`:

- **`Main.kt`** — wires everything. `tools = listOf(...)` is the authoritative tool registry.
- **`http/KtorApp.kt`** — Ktor bootstrap; bearer check in `authorized()`; JSON-RPC pass-through. Notifications (no `id`) → `202 Accepted` no body (Streamable HTTP).
- **`mcp/McpServer.kt`** — JSON-RPC dispatcher; implements only `initialize`, `notifications/initialized`, `ping`, `tools/list`, `tools/call`. `PROTOCOL_VERSION = "2025-06-18"`. No SSE, no resources, no prompts.
- **`mcp/Tool.kt`** + **`mcp/tools/*Tool.kt`** — each declares `name`, `description`, JSON Schema `inputSchema`, `suspend execute() → JsonElement`. Dispatcher wraps return as one `text` content block (stringified JSON); errors → `isError: true` with `ERROR: <msg>`.
- **`krill/KrillRegistry.kt`** — one `KrillClient` per `config.seeds[]`. `bootstrap()` probes each with 5 retries / 2 s backoff (systemd startup race: krill-mcp comes up before Krill server is listening). `resolve()` does a single-shot re-probe if registry empty on call — this powers `reseed_servers` without shell access. Seed lookup: server id, host, or `host:port`.
- **`krill/KrillClient.kt`** — thin per-server HTTPS client. Trusts self-signed certs (Krill installs its own); security from PIN-derived bearer, not the cert. Exposes `/health`, `/nodes`, `/node/{id}`, `/node/{id}/data/series`, Project/Diagram writes. Returns raw `JsonElement` (deliberately doesn't mirror Krill's internal model).
- **`auth/PinProvider.kt`** — reads `pin_derived_key` lazily + re-readably; holds local `constantTimeEquals`.
- **`KrillClient.publicBaseUrl`** — derived at probe time from `/health.meta` (name + port; `.local` suffixed for bare/mDNS names). Used for `Diagram.source` URLs so LAN phones/browsers can fetch the SVG (`localhost:8442` would be unreachable elsewhere).

### PIN derivation — wire contract, do not change

Bearer = `HMAC-SHA256(key="krill-api-pbkdf2-v1", data=PIN)` → hex, via `krill-sdk` `krill.zone.shared.security.PinDerivation.deriveBearerToken`. **Must stay byte-identical** with the `openssl dgst` line in `package/DEBIAN/postinst` and the upstream Krill server's postinst. SDK constant drift breaks auth swarm-wide. The HMAC key (`"krill-api-pbkdf2-v1"`) is **not** versioned.

### Adding an MCP tool

1. Implement `Tool` in `mcp/tools/` (pattern: `*Tool.kt` — `KrillTools.kt` for reads, `DiagramTools.kt` for Project/Diagram writes; both are multi-class files).
2. Register in `Main.kt`'s `tools = listOf(...)`.
3. Update both `skill/krill/SKILL.md` and `skill/krill/references/mcp-tools.md` in the same PR or skill docs drift.
4. Return `JsonObject` / `JsonArray` (not a string) — values are stringified verbatim into the text content block, so Claude needs structured data.

### Version sync — five sites

Every `krill-mcp` release bumps version in five places (drift = misleading headers, wrong skill behavior, `apt install` that doesn't overwrite):

1. `krill-mcp-service/build.gradle.kts` → `version = "X.Y.Z"`
2. `krill-mcp-service/src/main/kotlin/krill/zone/mcp/Main.kt` → `SERVER_VERSION = "X.Y.Z"`
3. `krill-mcp-service/package/DEBIAN/control` → `Version: X.Y.Z`
4. `skill/krill/SKILL.md` frontmatter → `version: X.Y.Z`
5. Every `vX.Y.Z` literal in `skill/krill/SKILL.md` + `skill/krill/references/mcp-tools.md` (`grep -rn 'v0\.0\.' skill/`).

Bug-fix PRs **do not** bump krill-mcp version — releases follow this sync + ship the .deb separately.

### Distribution

Debian package (`package/` = staged `dpkg-deb` tree). `postinst` creates the `krill` system user, symlinks `/etc/krill-mcp/credentials/pin_derived_key` to an existing Krill install's key **or** prompts for a 4-digit PIN, writes `/etc/systemd/system/krill-mcp.service`, prints bearer + Custom Connector URL. `Architecture: all` (pure JVM). The `.deb` itself is built by the private `krill` repo's "Deploy Debian Repo" workflow.

### Companion skill (`skill/krill/`)

Documentation-only Claude skill teaching MCP tool usage + SVG dashboard authoring. Bundles 37 Krill node-type JSON specs copied from `krill`'s `shared/src/commonMain/resources/`. Local install: `ln -s "$(pwd)/skill/krill" ~/.claude/skills/krill`. Touching MCP tool shapes? Update `skill/krill/references/mcp-tools.md` in the same PR.

## `pi4j-ktx-service/` — JDK 21 (lib) + JDK 25 (daemon)

Daemon pins `jvmToolchain(25)`; client lib pins `jvmToolchain(21)`. **Intentional — don't unify** (Pi4J runs on Pi under JDK 25; consumers stay on JDK 21). Bug-fix PRs don't bump version; releases coordinate with the upstream Pi4J Maven publish.
