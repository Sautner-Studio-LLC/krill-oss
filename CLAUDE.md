## Routing an upstream root cause (dev ‚Üí dev)

When a dev agent's investigation bottoms out in the **other** repo:

1. **Don't fix it in the current repo.** Leave the originating issue open; never patch around an upstream bug unless QA explicitly accepts a downstream mitigation, and then only with a tracker issue describing the workaround.
2. **File the upstream issue** using `gh issue create --repo <other-repo> ... --assignee krill-blue-bot` with **labels passed explicitly**. The `--assignee` is what gets the upstream dev to actually work it; without it the issue sits in nobody's queue. Reuse the originating QA report's body shape so the upstream dev sees what they'd see from QA directly.
3. **Cross-link both ways.** Comment on the originating issue: *"Routed upstream as `<other-repo>#<m>`. Leaving this open until that lands."* On the upstream issue body, add an `Observed downstream:` line pointing at the originating issue.
4. **`dev-krill` only:** open a tracker issue here with `dev-cross-repo.yml` so the originating QA issue stays linked to upstream work that lives on the other repo.
5. **Wait.** Only the QA agent closes QA issues, after re-verifying once the upstream fix lands.

## Per-issue work loop

1. **Reproduce locally first.** If you can't reproduce after a real attempt, comment on the issue asking Ghost for the missing detail, then stop. Do not fix what you cannot reproduce. (`dev-krill-agents` exception: most issues against `krill-agents` are documentation/workflow gaps with no "reproducer" beyond reading the file ‚Äî see `roles/dev-krill-agents.md`.)
2. **Locate the regression.** `git log -S '<distinctive token>'` and `git blame` on the suspect file. Capture the introducing commit SHA + title for the PR body. If the bug predates the repo or is genuinely greenfield, say so explicitly.
3. **Branch.** `git checkout -b fix/<issue#>-<slug>` from `main`. Slug = 3‚Äì5 hyphen-separated words from the issue title. Use `feat/<slug>` only when Ben asks for a feature; `chore/<slug>` for repo plumbing or doc/workflow updates on `krill-agents`.
4. **Test-first.** Write the failing test that captures the regression, then make it pass. Confirm red ‚Üí green. If the bug is genuinely untestable (UI rendering on a real device, hardware on Pi), say so explicitly in the PR ‚Äî do not paper over with a token assertion. For doc-only fixes, prefer a markdown sanity test (e.g. assert the absolute phrasing is gone) so the regression has a guard. (`dev-krill-agents` exception: this repo has no test framework; verify by running `scripts/build-claude-md.sh <role> -` and reading the output as a fresh agent would.)

## Writing tests that don't flake

CI runs on hardware that doubles as a real Krill server: filesystem paths the production code expects (`/etc/krill/...`, `~/.krill/install_id`, `/var/lib/krill/...`, `/srv/krill/...`) actually exist, with real PIN files and install ids. Wall-clock-based assertions live next to background traffic. Anything that bakes assumptions about either is a flake waiting to happen ‚Äî and CI treats it as a hard failure.

The rules below are absolute for any new test landing in a fix PR. If you can't satisfy them, the unit-under-test is wrong-shaped ‚Äî either parameterise the seam (preferred) or skip the test and explain why in the PR.

### Bans (hard fails ‚Äî delete or rewrite the test)

- **No `delay()` (or `Thread.sleep`) to simulate time passing.** A test that fires N coroutines and waits 250 ms expecting "the first one wins" depends on `Dispatchers.Default`'s scheduling, which CI doesn't owe you. Use `runTest { ‚Ä¶ }` from `kotlinx-coroutines-test` and `advanceTimeBy(...)` / `advanceUntilIdle()` ‚Äî virtual time, deterministic. The library is in `gradle/libs.versions.toml` as `kotlin-coroutines-test` but **not yet wired into every module's test classpath**; if the import fails, add `testImplementation(libs.kotlin.coroutines.test)` to the module's `build.gradle.kts` in the same PR.
- **No wall-clock-relative assertions in concurrency tests.** "Fired exactly once after a 250 ms window" is a timing race, not a correctness test. If the production code has a mutex/serialisation invariant, assert it without time ‚Äî for example, count concurrent enters via a debug-only `Semaphore.tryAcquire()` probe in the production hot path, or feed the function ordered events through a virtual scheduler.
- **No reading or writing real production paths.** `/etc/krill/credentials/pin_derived_key`, `/srv/krill/data/krill_v3.db`, `~/.krill/install_id`, `/var/lib/krill/trusted` and their cousins are owned by deployed Krill instances. CI runners co-host real ones. Test fixtures that depend on those paths being absent fail on the runner; fixtures that write to them clobber a real install. Inject the path (or the value the path resolves to) as a constructor / function parameter and pass a fixture string in the test.
- **No network calls, even to localhost.** That includes Ktor `testApplication { ‚Ä¶ }` if it binds a real port. Use `MockEngine` from `io.ktor.client.engine.mock` against the route handler factory directly, or call the helper the route delegates to.
- **No tests that depend on `installId()`, `hostName`, `SystemInfo.isServer()`, or any other platform `expect-val`.** These read the environment + filesystem on first call. The production code may legitimately need them; the test should not. If your unit-under-test calls `installId()` internally, refactor it to take the id as a parameter and let the caller (route handler, processor `init`) resolve it once at the boundary. `Sautner-Studio-LLC/krill#190`'s `peerProjection(nodes, selfId)` is the canonical shape ‚Äî pure helper, deterministic test, no `@BeforeEach` `mkdirs`.
- **No `@BeforeEach { File(somePath).parentFile?.mkdirs() }` or any other "make the environment look right before this test" setup.** That's the symptom of a missing seam. Fix the seam.

### Positive patterns

- **Parameterise the platform seam.** Helper takes `selfId: String` (or `now: () -> Long`, or `clock: Clock`); the route handler resolves the real value once and passes it in. Tests pass a literal.
- **Stub via `open` + subclass.** `PinProvider` is `open`; tests subclass with `override fun bearerToken(): String? = "fixed"`. Same shape applies anywhere the unit-under-test exposes one I/O surface.
- **Virtual time over real time.** `runTest { advanceTimeBy(150.milliseconds); coroutine.cancelAndJoin() }` is deterministic at any CI load.
- **`MockK` for collaborator boundaries, not for the unit under test.** Mock `ServerNodeManager`, not the function you're testing.
- **Prove it with a hostile environment.** Before pushing, run the test once with `HOME=/nonexistent ./gradlew :module:test`. If it still passes, the test is environment-independent. PR #190 uses this as the verification step in its commit message.

If a test legitimately needs a specific environment to be meaningful (integration, hardware-on-Pi), it doesn't belong in `:server:jvmTest` / `:shared:test` / etc. Mark it explicitly in the PR body, put it behind a JUnit tag the default `:test` task skips, or skip writing it and flag the gap.

### Background ‚Äî why this section exists

`Sautner-Studio-LLC/krill#190` (the `/peers` route fix) had to delete three pre-existing tests for being environment-dependent: two `PinProviderTest` cases that assumed `/etc/krill/credentials/pin_derived_key` does not exist on the test host (it does, on the runner), and one `SilentAlarmWatchdogTaskTest` case that fired 20 concurrent launches and asserted single-fire after a 250 ms wall-clock delay. Don't recreate any of those shapes ‚Äî and if you find existing ones in the codebase, prefer deleting them over fixing them up unless the underlying invariant is genuinely worth covering, in which case rewrite to virtual time + a parameterised seam.

## `krill-mcp/` ‚Äî JDK 21, JVM-only

```bash
cd krill-mcp
./gradlew :krill-mcp-service:shadowJar                   # Fat JAR ‚Üí krill-mcp-service/build/libs/krill-mcp-all.jar
./gradlew :krill-mcp-service:build                       # Build (depends on shadowJar) + tests
./gradlew :krill-mcp-service:test --tests "FQCN.method"  # Single test
java -jar krill-mcp-service/build/libs/krill-mcp-all.jar # Run; needs /etc/krill-mcp/config.json + credentials
```

`./gradlew :krill-mcp-service:run` is **not wired** ‚Äî run the fat JAR directly.

The daemon expects:

- `/etc/krill-mcp/config.json` ‚Äî `KrillMcpConfig` (`listenPort`, `pinDerivedKeyPath`, `seeds[]`).
- `/etc/krill-mcp/credentials/pin_derived_key` ‚Äî 64-char hex HMAC of the cluster PIN.

Without a PIN-derived key the server boots but rejects every `/mcp` request as `401`. `/healthz` stays unauthenticated.

### Architecture

```
Claude ‚îÄ‚îÄBearer‚îÄ‚îÄ‚ñ∂ KtorApp /mcp ‚îÄ‚îÄ‚ñ∂ McpServer ‚îÄ‚îÄ‚ñ∂ Tool.execute() ‚îÄ‚îÄ‚ñ∂ KrillRegistry.resolve() ‚îÄ‚îÄ‚ñ∂ KrillClient ‚îÄ‚îÄHTTPS‚îÄ‚îÄ‚ñ∂ Krill server
```

Key modules under `krill-mcp-service/src/main/kotlin/krill/zone/mcp/`:

- **`Main.kt`** ‚Äî wires everything. The `tools = listOf(...)` block is the authoritative tool registry; adding a new MCP tool means adding a `Tool` impl and listing it here.
- **`http/KtorApp.kt`** ‚Äî Ktor bootstrap. Bearer check in `authorized()`; everything else is plain JSON-RPC pass-through. Notifications (no `id`) respond `202 Accepted` with no body, per Streamable HTTP.
- **`mcp/McpServer.kt`** ‚Äî JSON-RPC dispatcher. Implements only the subset Claude clients need: `initialize`, `notifications/initialized`, `ping`, `tools/list`, `tools/call`. `PROTOCOL_VERSION = "2025-06-18"`. No SSE streaming, no resources, no prompts.
- **`mcp/Tool.kt`** + **`mcp/tools/*Tool.kt`** ‚Äî each tool declares `name`, `description`, a JSON Schema `inputSchema`, and a `suspend execute()` returning any `JsonElement`. The dispatcher wraps the return as a single `text` content block (stringified JSON); errors become `isError: true` with `ERROR: <message>`.
- **`krill/KrillRegistry.kt`** ‚Äî owns one `KrillClient` per seed in `config.seeds[]`. `bootstrap()` probes each seed with 5 retries / 2s backoff to handle the **systemd startup race** where `krill-mcp` comes up before the Krill server it targets is listening. `resolve()` does a synchronous single-shot re-probe if the registry is still empty on a call ‚Äî this is what makes the `reseed_servers` MCP tool viable without shell access. Seed lookup accepts server id, host, or `host:port`.
- **`krill/KrillClient.kt`** ‚Äî thin HTTPS client per server. Trusts self-signed certs (Krill installs its own); security comes from the PIN-derived bearer, not the cert. Exposes `/health`, `/nodes`, `/node/{id}`, `/node/{id}/data/series`, plus Project/Diagram writes. Returns raw `JsonElement` trees ‚Äî deliberately *does not* mirror Krill's internal data model.
- **`auth/PinProvider.kt`** ‚Äî reads `pin_derived_key` from disk; lazy and re-readable. Holds the local `constantTimeEquals` used for bearer validation.
- **`KrillClient.publicBaseUrl`** ‚Äî derived at probe time from `/health.meta` (name + port, with `.local` suffixing for bare/mDNS names). Used when building `Diagram.source` URLs so phones/browsers on the LAN can fetch the SVG ‚Äî a `localhost:8442` seed would be unreachable from any other host.

### PIN derivation ‚Äî wire contract, do not change

The bearer is `HMAC-SHA256(key="krill-api-pbkdf2-v1", data=PIN)` ‚Üí hex, provided by the `krill-sdk` library at `krill.zone.shared.security.PinDerivation.deriveBearerToken`. **Must stay byte-identical** with the `openssl dgst` line in `package/DEBIAN/postinst` and the upstream Krill server's postinst. Any drift in the SDK constant breaks authentication swarm-wide. The HMAC key (`"krill-api-pbkdf2-v1"`) is **not** part of release versioning.

### Adding an MCP tool

1. Implement `Tool` in `mcp/tools/` (pattern: `*Tool.kt` in either `KrillTools.kt` for reads or `DiagramTools.kt` for Project/Diagram writes ‚Äî both are multi-class files).
2. Register it in `Main.kt`'s `tools = listOf(...)`.
3. Update **both** companion skill references ‚Äî `skill/krill/SKILL.md` and `skill/krill/references/mcp-tools.md` ‚Äî or the skill documentation drifts from reality.
4. Tool return values are stringified verbatim into a text content block; return a `JsonObject` / `JsonArray` so Claude gets structured data, not a string.

### Version sync ‚Äî five sites

Every `krill-mcp` release bumps version in five places; drift causes misleading headers, wrong skill behavior, or an `apt install` that doesn't overwrite. Full list:

1. `krill-mcp-service/build.gradle.kts` ‚Üí `version = "X.Y.Z"`
2. `krill-mcp-service/src/main/kotlin/krill/zone/mcp/Main.kt` ‚Üí `SERVER_VERSION = "X.Y.Z"`
3. `krill-mcp-service/package/DEBIAN/control` ‚Üí `Version: X.Y.Z`
4. `skill/krill/SKILL.md` ‚Üí frontmatter `version: X.Y.Z`
5. Every `vX.Y.Z` literal in `skill/krill/SKILL.md` and `skill/krill/references/mcp-tools.md` (`grep -rn 'v0\.0\.' skill/` catches them).

Bug-fix PRs **do not** bump krill-mcp version. Releases follow this five-site sync and ship the Debian package; treat that as a separate, deliberate flow.

### Distribution

The daemon ships as a Debian package (`package/` is a staged `dpkg-deb` tree). `postinst` creates the `krill` system user, either symlinks `/etc/krill-mcp/credentials/pin_derived_key` to an existing Krill install's key **or** prompts interactively for a 4-digit PIN, writes `/etc/systemd/system/krill-mcp.service`, and prints the bearer token + Custom Connector URL. `Architecture: all` ‚Äî pure JVM, no native code. The `.deb` itself is built by the **private** `krill` repo's "Deploy Debian Repo" workflow.
