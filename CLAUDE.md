## Routing an upstream root cause (dev → dev)

When investigation bottoms out in the **other** repo:

1. **Don't fix locally.** Leave the originating issue open; only mitigate downstream if QA explicitly accepts it (with a tracker describing the workaround).
2. **File upstream:** `gh issue create --repo <other-repo> --assignee krill-blue-bot ...` with labels passed explicitly. **No `qa` / `qa-agent` umbrella** — that's QA's signal; `--assignee` is what gets the upstream dev. Reuse the originating QA report's body shape.
3. **Cross-link both ways:** comment on origin *"Routed upstream as `<repo>#<m>`. Leaving open until that lands."*; on upstream body add `Observed downstream:` pointing back.
4. **`dev-krill` only:** also open a tracker here via `dev-cross-repo.yml`.
5. **Wait** — only Ghost closes QA issues after re-verify.


## Acknowledge and report progress

Your work is asynchronous and the issue thread is the only place Ben (and Ghost, on the product repos) can see what you're doing. Treat it as a status log, not a silent workspace.

**On pick-up — before any fix work** — acknowledge the issue:

1. Apply the `status:in-progress` label so the issue list shows at a glance which tickets are actively in flight. If the label doesn't exist on the repo yet, create it once and re-use it from then on:

   ```bash
   gh label create status:in-progress \
     --repo <my-repo> \
     --color BFD4F2 \
     --description "Blue is actively working this issue"
   ```

   The label clears automatically when the merging PR closes the issue via `Closes #<n>`.
2. Comment with a one-line plan: *"Picking this up — \<one-sentence approach\>."* If you can't yet form an approach (e.g. you still need to read the code), say *"Picking this up — investigating reproducer."*

**During work** — comment at each meaningful inflection point so Ben can monitor progress without scrolling your shell logs. Comment when you:

- Locate the introducing commit (cite the SHA + title).
- Choose between two viable approaches and want to flag the call (e.g. *"going with X over Y because Z"*).
- Discover the root cause is upstream and you're routing it to a sibling repo (link the upstream issue both ways).
- Hit an unexpected blocker that will delay the fix beyond the current invocation, or that changes the scope.
- Open the PR (link it from the issue, in addition to `Closes #<n>` in the PR body).

You don't need to narrate every command — comments are for decisions, conclusions, and pivots, not running commentary. If a long invocation goes by with no comment between "picking this up" and the PR link, you've probably skipped a checkpoint.

## Per-issue work loop

1. **Reproduce locally first.** If you can't reproduce after a real attempt, comment on the issue asking Ghost for the missing detail, then stop. Do not fix what you cannot reproduce. (`dev-krill-agents` exception: most issues against `krill-agents` are documentation/workflow gaps with no "reproducer" beyond reading the file — see `roles/dev-krill-agents.md`.)
2. **Locate the regression.** `git log -S '<distinctive token>'` and `git blame` on the suspect file. Capture the introducing commit SHA + title for the PR body. If the bug predates the repo or is genuinely greenfield, say so explicitly.
3. **Branch.** `git checkout -b fix/<issue#>-<slug>` from `main`. Slug = 3–5 hyphen-separated words from the issue title. Use `feat/<slug>` only when Ben asks for a feature; `chore/<slug>` for repo plumbing or doc/workflow updates on `krill-agents`.
4. **Test-first.** Write the failing test that captures the regression, then make it pass. Confirm red → green. If the bug is genuinely untestable (UI rendering on a real device, hardware on Pi), say so explicitly in the PR — do not paper over with a token assertion. For doc-only fixes, prefer a markdown sanity test (e.g. assert the absolute phrasing is gone) so the regression has a guard. (`dev-krill-agents` exception: this repo has no test framework; verify by running `scripts/build-claude-md.sh <role> -` and reading the output as a fresh agent would.)
5. **Test location** (per repo): see `build-krill.md` and `build-krill-oss.md` for the right source set per module. `krill-agents` has no tests.
6. **Lessons entry** — `docs/lessons/YYYY-MM-DD-<slug>.md` on `krill` and `krill-oss`. Sections: *What happened* (one paragraph), *Fix* (bulleted, file paths included), *Prevention* (bulleted, generalizable rules — not a re-statement of the fix). Schema lives in `docs/lessons/README.md` of each repo. **Skipping this is a hard fail — both product repos' CI rejects PRs missing one.** `krill-agents` does not require lessons entries.
7. **Open the PR** using `.github/PULL_REQUEST_TEMPLATE.md` (Summary / Approach / Introducing commit / Test plan) with `Closes #<n>` in the body. Then decide whether to apply `needs-qa-verify` — see "When to apply `needs-qa-verify`" below. **On `krill-agents`, never add `needs-qa-verify`** — that repo has no QA verify workflow (it ships docs/scripts, not deployable code).
8. **Hand off (krill / krill-oss only).** If you applied `needs-qa-verify`, comment `@qa-agent please verify` with a precise reproduction script (see `cross-repo-protocol.md`); the label fires `Verify Agent Ghost.yml` and Ghost picks it up. If you skipped the label (no-risk PR), just `@` Ben for review and explain in one line why QA verify isn't needed (e.g. "doc-only — no runtime change"). On `krill-agents`, just `@` Ben for review.
9. **Stop.** On `krill` / `krill-oss`: wait for Ghost **PASS** + Ben's approval if the PR carried `needs-qa-verify`; otherwise wait for Ben's approval alone. On `krill-agents`, wait for Ben's approval.

### When to apply `needs-qa-verify` (krill / krill-oss only)

`needs-qa-verify` is the trigger for `Verify Agent Ghost.yml`, which builds all four debs, installs them on the ghost runner, and drives a full smoke + regression sweep through the swarm via the `krill` MCP service. That's a real cost — both wall-clock time and token budget — so only pay it when **(a) the PR could change runtime behavior** and **(b) Ghost can actually verify that behavior through MCP tool calls**. A Ghost run on a change Ghost can't observe through MCP is pure waste.

**Before labeling, answer this question:** *"Which specific `krill` MCP tool call would confirm that this fix works?"* If you cannot name one concretely, Ghost cannot verify the change — skip the label regardless of runtime risk.

**Apply `needs-qa-verify` when the PR could change MCP-observable runtime behavior.** That covers:

- Server-side logic changes under `server/` or `shared/` (krill) — visible via MCP node/state queries.
- `krill-sdk/` or `krill-mcp/server/` changes on `krill-oss` — MCP tool shape or SDK behavior that agents consume directly.
- MCP tool-shape or skill-instruction changes on `krill-oss` — agents in the field consume those at runtime.
- Build / Gradle / CI / packaging changes (`build.gradle.kts`, `gradle/libs.versions.toml`, `*.gradle.kts`, debian packaging, GitHub Actions YAML that builds or installs) — these affect what ships; Ghost's install smoke test catches regressions here.
- KSP processor changes — they regenerate code that affects the server's API surface, which is MCP-observable.

**Skip `needs-qa-verify` when the PR has no runtime risk, or when the runtime effect is not MCP-observable.** That covers:

- Pure documentation: `README.md`, `CLAUDE.md`, `docs/**` (including `docs/lessons/` entries on their own), in-repo `*.md` under module roots that aren't loaded at runtime.
- Comment-only edits in source files (no behavior change).
- `openspec/` proposals and design docs.
- Repo plumbing that the build doesn't consume (e.g. `.gitignore`, `.editorconfig`, `CODEOWNERS`, issue/PR templates under `.github/ISSUE_TEMPLATE/` or `.github/PULL_REQUEST_TEMPLATE.md`).
- **UI-only changes** in `composeApp/`, `androidApp/`, or `iosApp/` whose effect is purely visual — layout, animations, Compose composables, UI interactions. Ghost verifies via MCP API calls, not by inspecting rendered UI. If a UI change also moves a server route or shared model, the server-side part may be MCP-verifiable — label only for that reason, and name the MCP tool call in the PR body.
- **Hardware-only changes** in `krill-pi4j/` affecting physical GPIO/PWM/I2C pin behavior with no corresponding observable change in MCP tool responses. Ghost cannot read physical pin voltages through MCP.
- **Test-seam-only parameterization:** when the sole production-code change is adding a constructor or function parameter with a default that keeps all existing call sites identical (e.g., `fun Foo(provider: () -> String = ::realValue)`), and no production DI wiring was added or changed, the runtime behavior is identical to the unmodified code. These changes exist solely to enable deterministic unit tests. They are not MCP-observable — skip regardless of module.

If a single PR mixes a code fix with its `docs/lessons/` entry, the code fix drives the decision — apply the label. The label is judged at PR scope, not per-file.

State your call in the PR body's Test plan section with one of:
- "Labeled `needs-qa-verify` — Ghost will verify via `<tool name>` call" (name the specific MCP tool).
- "Skipping `needs-qa-verify` — no runtime change (doc-only / comment-only / openspec)."
- "Skipping `needs-qa-verify` — runtime change but not MCP-verifiable (UI-only / hardware-only)."
- "Skipping `needs-qa-verify` — test-seam only (default preserves all existing call sites; production behavior unchanged)."

## Writing tests that don't flake

CI runs on hardware that doubles as a real Krill server: filesystem paths the production code expects (`/etc/krill/...`, `~/.krill/install_id`, `/var/lib/krill/...`, `/srv/krill/...`) actually exist, with real PIN files and install ids. Wall-clock-based assertions live next to background traffic. Anything that bakes assumptions about either is a flake waiting to happen — and CI treats it as a hard failure.

The rules below are absolute for any new test landing in a fix PR. If you can't satisfy them, the unit-under-test is wrong-shaped — either parameterise the seam (preferred) or skip the test and explain why in the PR.

### Bans (hard fails — delete or rewrite the test)

- **No `delay()` (or `Thread.sleep`) to simulate time passing.** A test that fires N coroutines and waits 250 ms expecting "the first one wins" depends on `Dispatchers.Default`'s scheduling, which CI doesn't owe you. Use `runTest { … }` from `kotlinx-coroutines-test` and `advanceTimeBy(...)` / `advanceUntilIdle()` — virtual time, deterministic. The library is in `gradle/libs.versions.toml` as `kotlin-coroutines-test` but **not yet wired into every module's test classpath**; if the import fails, add `testImplementation(libs.kotlin.coroutines.test)` to the module's `build.gradle.kts` in the same PR.
- **No wall-clock-relative assertions in concurrency tests.** "Fired exactly once after a 250 ms window" is a timing race, not a correctness test. If the production code has a mutex/serialisation invariant, assert it without time — for example, count concurrent enters via a debug-only `Semaphore.tryAcquire()` probe in the production hot path, or feed the function ordered events through a virtual scheduler.
- **No reading or writing real production paths.** `/etc/krill/credentials/pin_derived_key`, `/srv/krill/data/krill_v3.db`, `~/.krill/install_id`, `/var/lib/krill/trusted` and their cousins are owned by deployed Krill instances. CI runners co-host real ones. Test fixtures that depend on those paths being absent fail on the runner; fixtures that write to them clobber a real install. Inject the path (or the value the path resolves to) as a constructor / function parameter and pass a fixture string in the test.
- **No network calls, even to localhost.** That includes Ktor `testApplication { … }` if it binds a real port. Use `MockEngine` from `io.ktor.client.engine.mock` against the route handler factory directly, or call the helper the route delegates to.
- **No tests that depend on `installId()`, `hostName`, `SystemInfo.isServer()`, or any other platform `expect-val`.** These read the environment + filesystem on first call. The production code may legitimately need them; the test should not. If your unit-under-test calls `installId()` internally, refactor it to take the id as a parameter and let the caller (route handler, processor `init`) resolve it once at the boundary. `Sautner-Studio-LLC/krill#190`'s `peerProjection(nodes, selfId)` is the canonical shape — pure helper, deterministic test, no `@BeforeEach` `mkdirs`.
- **No `@BeforeEach { File(somePath).parentFile?.mkdirs() }` or any other "make the environment look right before this test" setup.** That's the symptom of a missing seam. Fix the seam.

### Positive patterns

- **Parameterise the platform seam.** Helper takes `selfId: String` (or `now: () -> Long`, or `clock: Clock`); the route handler resolves the real value once and passes it in. Tests pass a literal.
- **Stub via `open` + subclass.** `PinProvider` is `open`; tests subclass with `override fun bearerToken(): String? = "fixed"`. Same shape applies anywhere the unit-under-test exposes one I/O surface.
- **Virtual time over real time.** `runTest { advanceTimeBy(150.milliseconds); coroutine.cancelAndJoin() }` is deterministic at any CI load.
- **`MockK` for collaborator boundaries, not for the unit under test.** Mock `ServerNodeManager`, not the function you're testing.
- **Prove it with a hostile environment.** Before pushing, run the test once with `HOME=/nonexistent ./gradlew :module:test`. If it still passes, the test is environment-independent. PR #190 uses this as the verification step in its commit message.

If a test legitimately needs a specific environment to be meaningful (integration, hardware-on-Pi), it doesn't belong in `:server:jvmTest` / `:shared:test` / etc. Mark it explicitly in the PR body, put it behind a JUnit tag the default `:test` task skips, or skip writing it and flag the gap.

### Background — why this section exists

`Sautner-Studio-LLC/krill#190` (the `/peers` route fix) had to delete three pre-existing tests for being environment-dependent: two `PinProviderTest` cases that assumed `/etc/krill/credentials/pin_derived_key` does not exist on the test host (it does, on the runner), and one `SilentAlarmWatchdogTaskTest` case that fired 20 concurrent launches and asserted single-fire after a 250 ms wall-clock delay. Don't recreate any of those shapes — and if you find existing ones in the codebase, prefer deleting them over fixing them up unless the underlying invariant is genuinely worth covering, in which case rewrite to virtual time + a parameterised seam.

## `krill-mcp/` — JDK 21, JVM-only

Daemon expects `/etc/krill-mcp/config.json` (`KrillMcpConfig`: `listenPort`, `pinDerivedKeyPath`, `seeds[]`) + `/etc/krill-mcp/credentials/pin_derived_key` (64-hex HMAC of cluster PIN). Without the key, server boots but `/mcp` returns 401; `/healthz` stays unauthenticated.

### Architecture

```
Claude ──Bearer──▶ KtorApp /mcp ──▶ McpServer ──▶ Tool.execute() ──▶ KrillRegistry.resolve() ──▶ KrillClient ──HTTPS──▶ Krill server
```

Modules under `krill-mcp-service/src/main/kotlin/krill/zone/mcp/`:

- **`Main.kt`** — wires everything. `tools = listOf(...)` is the authoritative tool registry.
- **`http/KtorApp.kt`** — Ktor bootstrap; bearer check in `authorized()`; JSON-RPC pass-through. Notifications (no `id`) → `202 Accepted` no body (Streamable HTTP).
- **`mcp/McpServer.kt`** — JSON-RPC dispatcher; implements only `initialize`, `notifications/initialized`, `ping`, `tools/list`, `tools/call`. `PROTOCOL_VERSION = "2025-06-18"`. No SSE, no resources, no prompts.
- **`mcp/Tool.kt`** + **`mcp/tools/*Tool.kt`** — each declares `name`, `description`, JSON Schema `inputSchema`, `suspend execute() → JsonElement`. Dispatcher wraps return as one `text` content block (stringified JSON); errors → `isError: true` with `ERROR: <msg>`.
- **`krill/KrillRegistry.kt`** — one `KrillClient` per `config.seeds[]`. `bootstrap()` probes each with 5 retries / 2 s backoff (systemd startup race: krill-mcp comes up before Krill server is listening). `resolve()` does a single-shot re-probe if registry empty on call — this powers `reseed_servers` without shell access. Seed lookup: server id, host, or `host:port`.
- **`krill/KrillClient.kt`** — thin per-server HTTPS client. Trusts self-signed certs (Krill installs its own); security from PIN-derived bearer, not the cert. Exposes `/health`, `/nodes`, `/node/{id}`, `/node/{id}/data/series`, Project/Diagram writes. Returns raw `JsonElement` (deliberately doesn't mirror Krill's internal model).
- **`auth/PinProvider.kt`** — reads `pin_derived_key` lazily + re-readably; holds local `constantTimeEquals`.
- **`KrillClient.publicBaseUrl`** — derived at probe time from `/health.meta` (name + port; `.local` suffixed for bare/mDNS names). Used for `Diagram.source` URLs so LAN phones/browsers can fetch the SVG (`localhost:8442` would be unreachable elsewhere).

### PIN derivation — wire contract, do not change

Bearer = `HMAC-SHA256(key="krill-api-pbkdf2-v1", data=PIN)` → hex, via `krill-sdk` `krill.zone.shared.security.PinDerivation.deriveBearerToken`. **Must stay byte-identical** with the `openssl dgst` line in `package/DEBIAN/postinst` and the upstream Krill server's postinst. SDK constant drift breaks auth swarm-wide. The HMAC key (`"krill-api-pbkdf2-v1"`) is **not** versioned.

### Adding an MCP tool

1. Implement `Tool` in `mcp/tools/` (pattern: `*Tool.kt` — `KrillTools.kt` for reads, `DiagramTools.kt` for Project/Diagram writes; both are multi-class files).
2. Register in `Main.kt`'s `tools = listOf(...)`.
3. Update both `skill/krill/SKILL.md` and `skill/krill/references/mcp-tools.md` in the same PR or skill docs drift.
4. Return `JsonObject` / `JsonArray` (not a string) — values are stringified verbatim into the text content block, so Claude needs structured data.

### Version sync — five sites

Every `krill-mcp` release bumps version in five places (drift = misleading headers, wrong skill behavior, `apt install` that doesn't overwrite):

1. `krill-mcp-service/build.gradle.kts` → `version = "X.Y.Z"`
2. `krill-mcp-service/src/main/kotlin/krill/zone/mcp/Main.kt` → `SERVER_VERSION = "X.Y.Z"`
3. `krill-mcp-service/package/DEBIAN/control` → `Version: X.Y.Z`
4. `skill/krill/SKILL.md` frontmatter → `version: X.Y.Z`
5. Every `vX.Y.Z` literal in `skill/krill/SKILL.md` + `skill/krill/references/mcp-tools.md` (`grep -rn 'v0\.0\.' skill/`).

Bug-fix PRs **do not** bump krill-mcp version — releases follow this sync + ship the .deb separately.
